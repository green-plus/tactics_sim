# 素数大富豪 戦術シミュレーションビューア

`primeqk_simulation_browser` で作った policy 結果JSONを読み、ブラウザ上で戦術別の採用率、期待勝率、試行例を確認する静的ビューアです。ブラウザ内で素数・合成数セットを入力し、新しい初手シミュレーションを実行することもできます。

## ローカル起動

`file://` ではJSON読み込みやWorkerが制限される場合があるため、ローカルサーバーから開きます。

```powershell
python -m http.server 8000 -d web
```

その後、`http://localhost:8000/` を開いてください。

## 収録データ

- `data/policy_study_300.json`: 通常サンプルの300試行レポート
- `data/policy_study_300_uta_sakura.json`: 追加セット入りサンプルの300試行レポート
- `data/win_rates.json`: 勝率モデルの初期値
- `data/sample_memory.json`: ブラウザ内シミュレーション用のサンプル素数・合成数セット

画面上部のタブは、初期状態では収録済みの `通常` / `追加` サンプルを表示します。新規シミュレーションを実行した後は、同じタブがユーザー入力から作った結果を表示します。元の収録データに戻すには、集計パネル右上の再読み込みボタンを押してください。

## 入力欄の使い方

`記憶セット` で、通常セット用の素数リストと合成数リストを入力します。

`追加セットを使う` をオンにすると、追加素数リストと追加合成数リストの入力欄が有効になります。この状態で実行すると、次の2種類を同じ条件でシミュレーションします。

- `通常`: 通常の素数リスト + 通常の合成数リスト
- `追加`: 通常の素数リスト + 通常の合成数リスト + 追加素数リスト + 追加合成数リスト

追加セットはUTASAKURA専用ではありません。サンプルではUTASAKURA由来の追加合成数が入っていますが、実装上は単に「追加欄に入っている行を通常セットへ足す」だけです。

## 素数リストの形式

素数リストは、空白、改行、カンマなどで区切られたカード表記を受け付けます。読み取り対象になる文字は `0-9`, `t`, `j`, `q`, `k` です。大文字小文字は区別しません。

- `t`, `j`, `q`, `k` はそれぞれ `10`, `11`, `12`, `13` として数値化されます。
- `0` は単独カードではなく、`10` を数字だけで書くときの一部です。ジョーカーではありません。
- ジョーカーはシミュレーション結果の出し方では `X` と表示されますが、素数・合成数リストの入力では受け付けません。
- 入力された値はBigIntのMiller-Rabin判定で素数か確認されます。素数でない値は警告になり、候補から除外されます。
- 物理カード表現に展開した結果、`1` のカードが5枚以上必要になる素数表現は除外します。ジョーカーを何枚も1として使う前提になり、実戦で検討する頻度が低いためです。

例:

```text
2
13
qk
1213
23456789
```

`qk` は数値としては `1213` です。内部では数値 `1213` から、実際に手札で作れるカード列を展開します。そのため `1213` は `qk`, `q13`, `12k`, `1213` のような候補を持ちます。サンプルの素数リストは主に分解後の数値表記ですが、ユーザー入力では絵札表記を使っても構いません。

## 合成数リストの形式

合成数リストは、基本的に次の形式です。

```text
左辺=右辺
```

例:

```text
256=2^8
kq=2^5*41
8qj=6247*k | 24678jqk
```

左辺は場に出す合成数です。右辺はその説明や消費カード候補として使うトークン列です。現在のブラウザ実装は、式を代数的に評価して検算するのではなく、`0-9tjqk` の連続部分をカード候補として取り出します。`*` や `^` は区切り記号として使われ、カード列そのものにはなりません。

例として `256=2^8` は、左辺 `256` と右辺の `2`, `8` をカード候補として使います。`^` を指数計算として解くのではなく、記法として分解している点に注意してください。

合成数でも、絵札と数値表記は候補展開されます。たとえば `kq=2^5*41` の左辺 `kq` は数値 `1312` で、内部候補として `kq`, `k12`, `13q`, `1312` などが生成されます。つまり、通常の式形式では `K` と `13` の両方の出し方が候補になります。

### `|` による消費カード固定

合成式の末尾に `|` を書くと、右側を「実際に消費するカード列」として固定できます。

```text
8qj=6247*k | 24678jqk
```

この場合、合成数として場に出す枚数は左辺 `8qj` の3枚ですが、手札から消費するカードは `24678jqk` に固定されます。`|` 以降は候補展開ではなく、指定した物理カード列をそのまま使います。特定の追加合成数セットを正確に再現したい場合に使います。

## 勝率モデル

`勝率モデル` では、候補戦術ごとの有効/無効と勝率上書きを変更できます。

- 数値欄が空なら、結果JSONや詳細勝率JSONに含まれる既定勝率を使います。
- 数値欄に `90` のように入れると、その戦術候補を一律90%として再評価します。
- `all-out` は最終フォールバックなので無効化できませんが、勝率は変更できます。
- `詳細勝率JSON` では `strategy_rates`, `conditional_rates`, `move_overrides` を直接編集できます。

戦術の選択肢は結果JSON内の候補から自動検出されます。ビューア側では、特定の追加セット専用の戦術制限は設けていません。

画面上では、次の日本語名を併記します。

| 内部ID | 表示名 |
| --- | --- |
| `x-lock` | Xロック |
| `dual-wield` | 二刀流 |
| `nine-plus` | 初手9枚出し以上 |
| `three-prime-eight-composite` | 3枚出し合成数上がり |
| `three-kkj` | KKJオーソドックス |
| `two-kk-factor` | KKオーソドックス |
| `one-x-one-x` | Xロック（1枚出し2回） |
| `face-five-prime` | 初手絵札5枚以上 |
| `trumps` | 初手切り札 |
| `all-out` | 全出し |

## 結果JSON

- `結果JSONを読む` で既存の policy レポートJSONを読み込めます。読み込んだJSONは `通常` タブのユーザー結果として表示されます。
- `表示中JSONをダウンロード` で、現在表示している結果を保存できます。勝率モデルを変更している場合は、表示中の再評価内容が反映されたJSONになります。
- ブラウザ内シミュレーションの乱数はJavaScript実装です。Python版と同じseedを入れても、完全に同一の試行列になる保証はありません。

## 現在の注意点

- ブラウザ内の合成式入力は、右辺の数式を厳密には検算しません。入力した式が数学的に正しいかは、必要に応じて元データ作成側で確認してください。
- 候補生成は最大カード枚数の範囲内で表記を展開します。非常に大きい値や表記の組み合わせが多い式は、候補数が増えて重くなることがあります。
- 追加セットをオンにして実行すると、通常と追加の2回シミュレーションするため、試行数に応じて実行時間も増えます。
